# PROJECT_RULES.md — AI Studio Vision Master Guidelines

This document contains the core architectural constraints, directory patterns, database schemas, and visual style tokens for **AI Studio Vision**. All subsequent features, files, and refactoring MUST strictly adhere to these rules.

---

## 1. ENVIRONMENTS & RUNTIME CONSTRAINTS

The platform runs in a containerized full-stack environment on Cloud Run.

### 1.1 Port & Networking
* **Port 3000** is the **ONLY** externally accessible port.
* Both frontend assets and backend endpoints MUST be served through a single Express + Vite middleware server (`server.ts`) listening on `0.0.0.0:3000`.

### 1.2 Hot Module Replacement (HMR)
* HMR is disabled (`DISABLE_HMR=true`) by the control plane.
* Avoid modifying Vite configs to force HMR. Code changes will be bundled incrementally; the workspace refreshes the preview automatically after each agent turn.

### 1.3 Strict ES Module Resolution
* All relative imports in backend or database TypeScript files MUST include full extensions (e.g., `import { db } from "./src/db/index.ts"`).
* Directory imports are **prohibited** (e.g., do NOT use `import { db } from "./src/db"`).

## 2. RECONCILED ARCHITECTURE (MASTER PROMPT VS. PORT CONSTRAINTS)

To satisfy the goals of `MASTER_01_PROJECT_ARCHITECTURE.md` within the AI Studio container layout:

```
[Client SPA (React 19)] 
         ↓ (Fetch API via standard HTTP / WS)
[Full-Stack Express Server (server.ts / port 3000)]
         ↓                               ↓
[Firebase Token verification]      [Gemini API / @google/genai SDK]
         ↓                               ↓
[PostgreSQL (Cloud SQL)]           [HuggingFace / Inference Layers]
         ↓
[Drizzle ORM Pool]
```

### 2.1 Reconciled Tech Stack Mapping
* **Frontend:** Next.js 15 **→ React 19 + Vite 6 + Tailwind CSS v4** SPA.
* **Backend:** Next.js Route Handlers **→ Express.js API Routes** integrated in `server.ts`.
* **Database & ORM:** Prisma **→ Drizzle ORM + pg (Pool)** connecting via unix socket/proxy.
* **Authentication:** Auth.js **→ Firebase Client-Side (signInWithPopup) + Firebase Admin SDK** server-side verification.
* **AI Service:** Python FastAPI **→ Express Services Layer** orchestration invoking Gemini models using the modern `@google/genai` TypeScript SDK.

## 4. DATABASE SYSTEM RULES (DRIZZLE & POSTGRESQL)

Drizzle ORM is **MANDATORY** for database schema representation.

### 4.1 Schema Guidelines (`src/db/schema.ts`)
* **Primary Identifier:** Relational tables MUST reference `users.uid` (the Firebase string `uid`) directly as keys, or execute explicit joins between integer user IDs and Firebase uids.
* **Soft Delete:** Include `deletedAt` timestamp columns on transactional records.
* **Indexes:** Define clear indexes on `uid`, `email`, `jobStatus`, and user lookups.

### 4.2 Error Isolation Rules (MANDATORY)
1. **Query Layer (Sanitized Exception):** Every database operation MUST be enclosed in a `try/catch` block. Catch raw Postgres/Drizzle errors, log them internally, and throw a sanitized message (e.g., `"Database query failed. Please try again later."`) to prevent table schemas or credentials from leaking.
2. **Caller Layer (UX Graceful Fallback):** The API controller must catch errors and return structured JSON (e.g., `{ error: "..." }`) rather than throwing an unhandled exception that causes a `500` blank page.

## 5. AUTHENTICATION & SECURITY RULES (FIREBASE AUTH)

We use Firebase Authentication exclusively, ensuring zero credential leaks.

### 5.1 Token Verification Workflow
1. Client signs in with `signInWithPopup(auth, googleAuthProvider)`.
2. Client fetches the JWT idToken using `await user.getIdToken()`.
3. Client attaches the ID Token in the request headers: `Authorization: Bearer <ID_TOKEN>`.
4. Express route handles authentication via `requireAuth` middleware using `adminAuth.verifyIdToken(token)`.
5. Dynamic user synchronization is done using Drizzle’s `onConflictDoUpdate` to prevent race conditions during concurrent user operations.

### 5.2 Key Security Policy
* **NO CLIENT-SIDE KEYS:** Never import `@google/genai` or instantiate GoogleGenAI clients on the browser.
* **LAZY INITIALIZATION:** All third-party clients (Firebase Admin, Drizzle Pool) MUST initialize lazily to avoid crashing the server on startup when keys are absent.
* **NO SECURITY DISCUSSION IN UI:** Never expose backend telemetry, database connection logs, or credentials in client UI cards.

## 6. DESIGN SYSTEM INTEGRATION (TAILWIND CSS V4)

Since **Tailwind CSS v4** is used (`@tailwindcss/vite` and `@import "tailwindcss";`), **there is no `tailwind.config.js`**. All typography, spacing, border-radius, color, and shadow system tokens MUST be written in the global stylesheet (`src/index.css`) using the `@theme` directive.

### 6.1 Theme Configuration Example (`src/index.css`)

## 7. AI INFERENCE & PROCESSING SYSTEM

For all AI generation tasks:
* **SDK:** Only use `@google/genai` TypeScript SDK with valid models (e.g., `'gemini-3.5-flash'` for basic/text orchestration, `'gemini-3.1-flash-lite-image'` for image operations).
* **Operation Flow:**
  1. Client sends file metadata to `/api/jobs/create`.
  2. Server responds with job token and initiates asynchronous background processing.
  3. Client polls `/api/jobs/status` until job state resolves to `SUCCESS` or `FAILED`.
  4. Client triggers download from `/api/jobs/download` (proxied server-side).
